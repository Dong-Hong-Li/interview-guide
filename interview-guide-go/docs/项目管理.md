# 项目管理（`interview-guide-go`）

> 与 `docs/项目架构.md` / `docs/开发规范.md` 配套；本文件覆盖**协作流程、分支、提交、PR、CI、版本、环境、发布、Issue、回滚**。  
> 没有专属 CI 时，以下「CI 检查」由 PR 作者本地 + Reviewer 双人复核执行。

---

## 1. 角色与责任

| 角色 | 职责 |
|------|------|
| Owner | 仓库治理、版本规划、合并主干；首位 Reviewer |
| Maintainer | 业务域接口扩展（简历/面试/知识库/RAG）；CR 必须有 Maintainer +1 |
| Contributor | 提 Issue / PR，本地自检通过；按照本文件流程提交 |
| Reviewer（轮值） | 24h 内首轮 review；阻塞超 48h 升级 Maintainer 介入 |

> 当前（仓库阶段）所有 PR 至少 1 名 Reviewer +1 + Owner 终审；面试评估 / 简历分析等关键链路必须 2 名 Reviewer。

## 2. 分支模型（精简 Trunk-Based + 短期特性分支）

### 2.1 分支命名

| 用途 | 命名 | 示例 |
|------|------|------|
| 主干 | `main` | 受保护，仅 PR 合并 |
| 发布候选 | `release/<MAJOR.MINOR.x>` | `release/0.1.x`（仅在准备 RC 时短期存在） |
| 特性 | `feat/<域>-<动词>-<细节>` | `feat/interview-list-pagination` |
| 修复 | `fix/<域>-<现象>` | `fix/interview-evaluate-context-canceled` |
| 重构 | `refactor/<范围>-<目的>` | `refactor/wire-resume-module-set` |
| 文档 | `docs/<topic>` | `docs/architecture-rewrite` |
| 杂活 | `chore/<topic>` | `chore/dockerfile-bump-go124` |
| 实验 | `spike/<topic>`（不允许合主干） | `spike/streaming-questions` |

### 2.2 主干约束

- `main` 始终可发布：`go build ./...` / `go vet ./...` / wire 重生成产物 / 文档计数一致。
- **禁止** `force-push` `main`；`release/*` 在合回 `main` 前不允许 rebase 历史。
- 分支生命周期 ≤ 7 天；超过强制拆分或与 `main` rebase。

### 2.3 发布分支

- 仅在做版本 cut 时从 `main` 切 `release/x.y.x`，只接收 cherry-pick 的 fix；
- 发布完成后回合 `main` 并打 tag（见 §6）。

## 3. 提交（Commit）规范

采用 [Conventional Commits](https://www.conventionalcommits.org/zh-hans/v1.0.0/) 中文 + 英文混排（仓库现有提交风格）。

### 3.1 格式

```
<type>(<scope>): <短描述，<= 60 chars>

<可选 body：动机 / 实现 / 影响>

<可选 footer：BREAKING CHANGE / 关联 Issue>
```

### 3.2 type 与 scope

| type | 用途 |
|------|------|
| `feat` | 新功能（含新端点 / 新服务） |
| `fix` | 缺陷修复 |
| `refactor` | 不改外部行为的内部重构 |
| `perf` | 性能改进 |
| `chore` | 构建、依赖、Docker、CI、脚手架 |
| `docs` | 仅文档变化 |
| `test` | 新增/修改测试 |
| `wire` | 仅 wire / DI 装配（含 `wire_gen.go` 重生成） |

scope 用域或层名：`interview` / `resume` / `kb` / `rag` / `infra` / `pdf` / `redis` / `pg` / `wire` / `cmd` / `docs` / `config` / `mw` / `binding`。

### 3.3 示例

```
feat(interview): 新增 GET /api/interview/sessions 分页列表

- 控制器 ListInterviewSessionsService.List 走 NormalizeListPaging
- Mapper.ListInterviewSessionsPage 按 created_at desc 返回 InterviewListItem 与 total
- 前端 history.ts 已对齐，文档同步日期 2026-04-24

Refs: #123
```

```
fix(interview): 修正 SubmitAnswer 在最后一题时未入队评估

最后一题写完未触发 evaluate_status=PENDING 与 enqueue.EnqueueInterviewEvaluate；
增加单测覆盖 hasNext=false 路径。

BREAKING CHANGE: 无
```

### 3.4 不允许的提交

- `WIP` / `tmp` / `aaa` 之类无意义信息；
- 单条提交跨多个 type（先拆 PR）；
- 大量重写历史文件但 message 仅说"调整"；
- 把 `wire_gen.go` 与业务变更混在同一 commit（拆成 `wire(server): regenerate after interview module change`）。

## 4. Pull Request

### 4.1 标题与正文

- 标题与首个 commit 一致（Conventional Commits）。
- 正文模板：

```markdown
## 背景 / 动机
（引用 Issue 或现象描述）

## 改动
- application/<domain>/...
- infrastructure/<...>/...
- shared/<...>/...

## 影响面
- API：是 / 否（列出新增/调整端点）
- DB：是 / 否（列出迁移）
- 配置：是 / 否（列出 ENV）

## 自检
- [ ] go build ./...
- [ ] go vet ./...
- [ ] wire ./cmd/server（如改了 wire.go）
- [ ] 端到端 happy path（贴 curl 或日志）
- [ ] 文档同步：docs/项目架构.md / docs/开发规范.md / docs/开发进度.md

## 其他
（已知风险、回滚方法、上线前后置依赖）
```

### 4.2 Reviewer 责任

- 24h 内首轮，2 个工作日内合并/驳回；
- 关注层界（application 不依赖 infrastructure）、端口与实现绑定、状态机门禁、错误/日志常量；
- 不阻塞性建议用 `nit:`，阻塞项写「必须」并给出最小修复路径。

### 4.3 合并方式

- 默认 **Squash & Merge**，标题保留 PR 标题，body 用最终 PR 描述；
- `wire` 与 `feat` 拆 PR 时如必须同 PR，使用 **Rebase & Merge** 保留 wire 单独的 commit；
- **禁止** Merge Commit（与默认 squash 冲突）。

## 5. 本地 / CI 自检

### 5.1 本地 pre-push

```bash
gofmt -l .
goimports -l .
go vet ./...
go build ./...
# 如改了 wire 标记的文件
( cd cmd/server && go run github.com/google/wire/cmd/wire@v0.7.0 . )
git diff --name-only -- 'cmd/server/wire_gen.go'  # 确认产物已 commit
```

### 5.2 推荐 CI（建仓后落地）

| 阶段 | 检查 | 失败处理 |
|------|------|---------|
| Lint | `gofmt -l . && goimports -l .` | 阻塞合并 |
| Vet | `go vet ./...` | 阻塞 |
| Build | `go build ./...` | 阻塞 |
| Test | `go test ./...`（unit）+ `go test -tags integration ./...`（带 db/redis） | 阻塞 |
| Wire 一致性 | 重跑 wire，diff `wire_gen.go`，非空则失败 | 阻塞 |
| Docker | `docker buildx build --platform linux/amd64 .` | 阻塞 |
| 文档同步 | grep `开发进度.md` 同步日期与 PR 日期一致（仅当 §1/§2 表变化） | 警告 |

> 当前若无 CI，Reviewer 必须要求作者贴出本地输出截图或粘贴文本。

## 6. 版本与发布

### 6.1 版本号

- 规则：[SemVer](https://semver.org/lang/zh-CN/) `MAJOR.MINOR.PATCH`；
- pre-release：`-rc.N`（候选）/ `-alpha.N`（早期）。

### 6.2 触发版本变更

| 变更 | 升 |
|------|------|
| 不兼容 API 变化（路径/方法/字段语义） | MAJOR |
| 新增端点 / 新增字段（向后兼容） / 新增配置（带默认值） | MINOR |
| 修复、重构、性能、依赖升级（无外部行为变化） | PATCH |

### 6.3 发布流程

```
1. 在 main 上 cut: git switch -c release/0.2.x
2. 跑 §5 全部检查；
3. 编写 CHANGELOG（按 type 聚合，版本号置顶，含日期）；
4. PR 合 release/0.2.x → main；
5. main 上打 tag: git tag -a v0.2.0 -m "release v0.2.0" && git push origin v0.2.0
6. 触发镜像构建（Dockerfile） + 推到镜像仓库（按部署平台）。
```

### 6.4 CHANGELOG 模板

```markdown
## v0.2.0 — 2026-MM-DD

### Added
- interview: GET /api/interview/sessions 分页列表（#xxx）

### Changed
- pdf: 中文字体提示在缺失时降级为英文水印（#xxx）

### Fixed
- interview: SubmitAnswer 最后一题未入队评估（#xxx）

### Internal
- wire: 重组 interviewModuleSet 减少重复绑定
```

## 7. 环境与配置基线

### 7.1 三套环境

| 环境 | 用途 | 数据 | LLM |
|------|------|------|-----|
| `dev`（本地） | 开发自测 | 本地 PG/Redis；可空对象存储 | `NewStubInterviewQuestionGenerator` 离线题（OPENAI_API_KEY 留空） |
| `staging` | 联调与回归 | 独立库；可清；OSS/S3 测试 bucket | 真 LLM，限额低（`AI_MODEL=gpt-4o-mini` 类） |
| `prod` | 生产 | 独立库；备份；正式 bucket | 正式 LLM；提高 `ResumeAITemperature` / token 上限按合同 |

### 7.2 ENV 清单（最小）

```
# Server
SERVER_HOST=0.0.0.0
SERVER_PORT=8080
SERVER_READ_TIMEOUT_SEC=120

# Postgres
DATABASE_URL=postgres://user:pass@host:5432/db?sslmode=disable

# Redis
REDIS_HOST=...
REDIS_PORT=6379
REDIS_PASSWORD=...
REDIS_DB=0

# Storage（OSS/S3 兼容）
STORAGE_PROVIDER=s3
S3_ENDPOINT=...
S3_REGION=...
S3_BUCKET=...
S3_ACCESS_KEY=...
S3_SECRET_KEY=...

# OpenAI 兼容
OPENAI_BASE_URL=https://api.openai.com/v1
OPENAI_API_KEY=...
AI_MODEL=gpt-4o-mini
RESUME_AI_MAX_RUNES=8000
```

## 8. 部署

### 8.1 镜像

- 单镜像两阶段构建（`Dockerfile`）：`golang:1.24-alpine` → `alpine:3.21`；
- 运行时安装 `ca-certificates`、`tzdata`、`poppler-utils`、`font-noto-cjk`；
- 非 root 用户 `appuser`（uid 10001）；
- 暴露 `EXPOSE 8080`；启动 `ENTRYPOINT ["/bin/server"]`。

### 8.2 部署形态

- 任意 K8s / docker-compose / 裸机 systemd 均可；
- **不要**部署多副本时让多个面试评估消费者抢同一组：消费者已通过 PID 区分 consumer name，但请确认 Redis Stream 上下游一致；
- 优雅停机：发送 SIGTERM，等 `httpServer.Shutdown(15s)` 完成 + 消费者 ctx.Done()。

### 8.3 配置注入

- ENV 优先；K8s 中用 `Secret` + `ConfigMap`；
- 不允许把 secret 提交进仓库；`.env*` 必须 `gitignore`；
- Docker Compose 联调时 `env_file` 引用本地未入库 `.env.local`。

### 8.4 健康检查

- Liveness：`GET /health`（站点 `system` 包，无依赖检查，永远 200）；
- Readiness：当前未单独提供；建议新增 `GET /api/meta/ready` 探测 PG/Redis（属于规范增量，需走 PR）。

## 9. 监控与排障

| 维度 | 数据来源 | 关键事件键 |
|------|---------|-----------|
| HTTP 访问 | `middleware.RequestLogger` | `method` / `path` / `status` / `duration` |
| 启动 | `cmd/server/main.go` + `config.LogStartup` | `MsgServerListening`、`MsgServerStopped` 等 |
| 简历分析消费者 | `redisstream.StartResumeAnalyzeConsumer` | `MsgResumeAIConsumerEnabled/Disabled` |
| 面试评估消费者 | `redisstream.StartInterviewEvaluateConsumer` | `MsgInterviewEvaluateConsumerStarted/Stopped` 等 |

排障 checklist：

1. 看启动日志确认 `MsgPostgresStartFailed/MsgRedisStartFailed/MsgOpenAIStartFailed` 是否出现；
2. 看消费者日志：`MsgInterviewEvaluateXReadGroup` 频繁 → 检查 Redis 连接；`MsgInterviewEvaluateLLMFailed` → 检查 OpenAI 配额/超时；
3. 看错误码：返回 503 → 注入项缺失（OpenAI/Redis 未就绪）；返回 409 → SETNX 锁冲突（前端在 90s 内重复创建）；返回 502 → LLM 上游错误。

## 10. Issue 与任务管理

### 10.1 Label

| Label | 含义 |
|-------|------|
| `bug` / `enhancement` / `refactor` / `chore` | 类型 |
| `area:interview` / `area:resume` / `area:kb` / `area:rag` / `area:infra` | 域 |
| `priority:p0` / `p1` / `p2` | 优先级 |
| `good first issue` | 适合新人 |
| `blocked` | 待外部依赖 |
| `breaking` | 不兼容变更 |

### 10.2 Issue 模板（建议）

```markdown
## 现象
（截图 / 报错 / 复现步骤）

## 期望
（含相关接口或文档链接）

## 复现环境
- 分支 / 版本：
- 配置：
- DB / Redis / OpenAI 是否就绪：

## 排查记录
（已查到什么、排除了什么）

## 建议方向
（可选，便于 PR 复用）
```

### 10.3 关联

- PR 描述里用 `Refs: #123` / `Closes: #123` 关联；
- Issue 关闭时附上落地 commit / PR 链接。

## 11. 文档治理

- 任何路由 / 端口 / 状态机 / 配置变更 → **同 PR** 改文档；
- `docs/开发进度.md` 第 6 行的「文档同步日期」由 PR 作者改成本次合并日；
- 文档冲突优先级：**代码 > `docs/开发进度.md` 表 > `docs/项目架构.md` > 主项目 Java 文档**（若与主项目冲突，先以本仓库代码为准、再补一句"与主项目差异说明"）。

---

## 12. 安全与合规

- 任何密钥（API Key、DB password、对象存储 secret）禁止入库；偶发泄漏立即旋转 + `git revert` + 强制重签 commit；
- LLM 输出落库前需要打分/过滤（参考 `interview_evaluator.go` 内的 prompt 防注入策略），不要把模型原始返回直接序列化进 Result.data；
- 上传文件大小受 `MAX_RESUME_UPLOAD_BYTES` 控制，禁止单测/灰度临时调高（必须走配置）；
- 删除接口（`DELETE /api/resumes/{id}` / `DELETE /api/interview/sessions/{id}`）默认硬删，慎重；如要软删，须在 PR 中明确并加列。

## 13. 回滚预案

| 故障 | 处置 |
|------|------|
| 镜像启动失败 / 配置错误 | 回滚到上一版镜像 tag；同时禁用最新 ConfigMap |
| 面试评估消费者持续 FAILED | 关停消费者进程（删除该 Deployment 副本） + 在 Redis 把对应 Stream 的待消费消息清理或转储；不删 `interview_sessions` 数据 |
| 简历上传写库成功但 LLM 长时间未回 | 维持现状，业务前端展示「分析中」；必要时手动 reanalyze |
| 数据库迁移导致字段缺失 | 回滚 SQL（必须有 down 脚本）；应用回滚到上一版本 |
| OpenAI 限额/欠费 | 切 `OPENAI_BASE_URL` 到备用兼容 endpoint，或临时注入 stub（重启时无 API_KEY 自动 stub） |

回滚后必须开 RCA Issue（`area:infra` + `priority:p0`），48h 内输出根因与防复发措施（参照规范 §3 提交流程）。

## 14. 沟通

- 紧急（P0/P1）：群内 @值班 + 同时开 Issue；
- 规划：每周一次 30 分钟同步会，议程从 Issue 看板拉取 `priority:p0/p1`；
- 大改动（涉及状态机 / 端口边界 / 新增基础设施）必须先发 Design Doc（放 `docs/design/<topic>.md`）并 review +1 后再开始实现。

---

## 15. 升级周期

- Go 主版本：每年随 Go 官方支持窗口升级（保留 N、N-1）；
- 关键依赖（`chi` / `gorm` / `go-redis` / `zap` / `openai-go` / `wire`）：每季度评估一次，PATCH 直接升、MINOR 评估、MAJOR 单 PR 单升；
- 自动化建议：开启 Renovate 或 Dependabot，但**不**自动合并 wire / gorm / redis 这类核心库。
