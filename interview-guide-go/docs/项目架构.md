# 项目架构（`interview-guide-go`）

> 本文件以**当前仓库代码**为唯一事实来源（不要套 Java 后端 `modules/*` 或 `router/router.go` 的目录习惯）。  
> 配合阅读：`docs/开发规范.md`、`docs/项目管理.md`、`docs/开发进度.md`。

---

## 1. 项目定位

- 模拟面试与简历分析的 Go 后端，与主项目 Java 后端的「面试域 + 简历域」能力对齐；知识库已部分实现；RAG（`/api/rag-chat`：会话 CRUD + **`POST .../messages/stream` SSE**）已实现。
- 单进程 HTTP 服务（`cmd/server`），出题与评估走 Redis Stream 异步消费者，PDF 在进程内同步生成。
- 关键中间件：PostgreSQL 16+（GORM）、Redis 7+（Streams + 字符串缓存 + SETNX 锁）、对象存储（S3/OSS 兼容）、OpenAI 兼容客户端。

```
│  ├─ domain/                 # 领域常量/纯函数门禁（无外设依赖）
│  ├─ infrastructure/         # 适配器：DB / Redis / Storage / AI / PDF / File
```

## 3. 分层与依赖方向（DDD 切片 + Hexagonal）

### 3.1 四层切片（自上而下，单向依赖）

```
interfaces (站点壳: 路由聚合、全局 mw、HTTP 绑定)
        ↓
application/<domain>/controller   ← HTTP 适配：绑定、**入参校验**、封装 `Validated*`、出参 Result
        ↓                              ↑（注入 services）
application/<domain>/service       ← 用例编排（单文件单用例）
        ↓                              ↑（依赖 repository 端口）
application/<domain>/repository    ← 端口（interface），不依赖任何具体实现
        ↑（实现）
infrastructure/<adapter>           ← Mapper（GORM）/ Redis adapter / Storage / AI / PDF
```

铁律：

- **`internal/application/*` 禁止 import `internal/infrastructure/*`**（除非 controller 层组装时由 deps 显式注入）。具体实现通过 `repository.*Port`（接口）反转。
- **`internal/domain/*` 禁止 import 应用层与基础设施**。只放纯函数 / 状态常量 / 门禁（如 `interview.CompleteInterviewGate`）。
- **`shared/*` 禁止 import `internal/*`**：错误文案、日志键、Stream 键、UUID、跨域状态字面量（`shared/interview/session_status.go`）等。
- **`internal/interfaces/*` 不写业务**，只做路由聚合、全局中间件、HTTP 绑定。
- **CORS**（`middleware/cors.go`，`go-chi/cors`）：须在环境中配置 **`CORS_ALLOWED_ORIGINS`**（逗号分隔；本地前端示例 `http://localhost:5173`）；未配置则 **`LoadEnvironmentVariables` 失败**。亦可设为 `*`（此时不携带 Cookie）。若通过 **局域网 IP** 访问 Vite（如 `http://192.168.2.1:5173`），须在 `CORS_ALLOWED_ORIGINS` 中写上该完整源，或追加 **`http://192.168.*:5173`** / **`http://10.*:5173`**，或设置 **`CORS_ALLOW_LAN_VITE_ORIGINS=1`**（由配置层合并上述两条通配；**生产勿开**）。
- **HTTP 入参规则**（必填、Trim、MIME/大小、multipart 正文抽取等）**只在 `application/.../controller` 完成**；`service` **不校验**前端原始请求体，只接收 `model.Validated*` 或领域层已接纳的入参（见 `docs/开发规范.md` §3.2、§4.3）。

## 4. 入口与依赖装配（`cmd/server`）

| 文件 | 角色 |
|------|------|
| `main.go` | 启动 logger → 加载 `config` → 调 `StartDeps` → 注册路由 → `http.Server` 启停（含 SIGINT/SIGTERM 优雅关停） |
| `deps.go` | **组合根**：起 Postgres / Redis / Storage / OpenAI；委托 wire 构造控制器；条件启动两个 Redis Stream 消费者；返回 `cleanup` |
| `wire.go` | wire 声明文件（`wireinject` build tag）。多个 ProviderSet（如 `resumeModuleSet` / `interviewModuleSet` / …）；接口与实现的 `wire.Bind` |
| `wire_gen.go` | `wire` 工具生成产物，**勿手改**；改完用 `wire ./cmd/server` 重新生成 |

启动顺序与跳过策略：

1. `logger.NewLogger()` 失败 → `log.Fatalf`（只在配置缺失时）；
2. `config.LoadEnvironmentVariables()` 任一段必填项缺失 → `log.Fatalf`；
3. Storage / Postgres / Redis / OpenAI 任一启动失败 → 仅 `lg.Error` 记录并 `return nil, cleanup`，HTTP 不会起来；
4. Redis Stream 消费者（简历分析 / 面试评估 / 知识库向量化）：`StartDeps` 已成功拉起 Postgres、Redis、OpenAI 等依赖后**必须**启动；前置不满足则 **`Fatal`**（禁止静默跳过）。面试出题依赖 **`provideInterviewQuestionGenerator`**：OpenAI 客户端不可用则 **panic**（无 Stub）。
5. **Postgres DDL**：`compose` 挂载的 `docker-entrypoint-initdb.d` **仅在空数据目录时**跑一遍；若本地 `pgdata` 卷早于某次 `schema/*.sql` 入库就存在，须在宿主机对**当前库**执行 **`internal/db/apply_schema.sh`**（`DATABASE_URL`）补表（常见缺表：`knowledge_base_chunks`）。见 **`internal/db/README.md`**。

## 5. HTTP 路由与请求生命周期

### 5.1 全局壳 `internal/interfaces/controller/router.go`

```text
chi.NewRouter()
  ├─ MountGlobal: chimw.RequestID / RealIP / Recoverer / RequestLogger(zap, suppress)
  └─ Route("/api"):
       ├─ MountAPIHooks: r.MethodNotAllowed = APIMethodNotAllowed (405 + Result)
       │                 r.NotFound        = APINotFound        (404 + Result)
       ├─ system.RootController.Register(r)            # GET /, GET /health, GET /api/meta
       └─ for ctrl in apiControllers: ctrl.Register(r) # 各域子路由
```

**业务接口统一挂在 `/api` 下，不使用 `/api/v1` 前缀**。各域控制器在 `Register` 内部 `r.Route(APIMountPath, ...)` 嵌套子路由。

### 5.2 单请求处理链路（以面试域为例）

```
HTTP → chi 全局 mw → /api 子路由
     → InterviewController.Register 注册的 chi 子树
     → binding.Handle[Req,Resp](fn) (泛型包装)
        ├─ bindRequest: JSON / multipart / path / query → Req
        ├─ Validate: 反射读 validate:"required"，零值即 400
        └─ fn(ctx, Req)  →  resp 或 *response.Error / *response.BizError
     → 成功: WriteJSON 200 + response.Result{code:200, message:"success", data:resp}
     → 失败: 按错误类型写 Result（HTTP 码与 body.code 对齐 / 业务级 200+code）
```

错误统一类型：

- `response.Err(code, msg)` → `*response.Error`：HTTP 状态码与 `Result.code` 一致。
- `response.BizErr(code, msg)` → `*response.BizError`：HTTP=200，但 `Result.code` 为业务码（用于客户端弱通道展示）。
- 其他 `error` → 500 + `errmsg.InternalServerError`。

### 5.3 PDF/二进制端点

简历导出 `GET /api/resumes/{id}/export` 与面试导出 `GET /api/interview/sessions/{sessionId}/export` **不**走 binding，直接用 `chi.URLParam` + `response.WriteErr` 处理；响应头 `Content-Type: application/pdf`、`Content-Disposition` 走 `pdfexport.ContentDispositionRFC5987`（中文文件名兼容）。

### 5.4 访问日志抑制

`config.parseHTTPAccessLogSuppress` 默认抑制：

- `GET /api/interview/sessions/*`（前端答题页轮询单会话）
- `GET /api/resumes/*/detail`
- `GET /api/knowledgebase/list`、`GET /api/knowledgebase/stats`、`GET /api/knowledgebase/categories`（知识库列表页三组轮询）

抑制规则在 `middleware.ShouldSuppressAccessLog` 用 `path.Match` 评估；新增高频轮询接口须同步加规则，避免日志噪声。

## 6. 业务域（截至本文件最后修订）

> 接口数量与覆盖率以 `docs/开发进度.md` 为准；本节只描述域内职责与边界。

### 6.1 简历域 `internal/application/resume`

- 控制器：`controller/controller.go` 注册 8 个端点（含 `POST /upload` multipart、`GET /export` PDF、`POST /{id}/reanalyze` 等）。
- 服务：上传分析、列表、详情、统计、重分析、导出 PDF、面试官角色枚举。
- 端口：`ResumeWriter`（落库）、`ObjectStorage`（OSS/S3）、`AnalyzeJobPublisher`（Redis Stream 入队）、`TextExtractor`（PDF/Doc 文本抽取）。
- 关键流程（上传分析）：
  1. `multipart/form-data` → `binding.bindMultipart` 填 `[]byte` 与文件元信息；
  2. `ResumeUploadService` 校验大小（`MAX_RESUME_UPLOAD_BYTES`）→ 抽取文本 → 落库 → 上传对象存储 → 入 `streamkey.StreamResumeAnalyze`；
  3. `redisstream.StartResumeAnalyzeConsumer` 消费、调 `ai.ResumeGrader` 写回分析结果。

### 6.2 面试域 `internal/application/interview`

- 控制器：`controller/controller.go` 注册 12 个端点（创建/列表/未完成/当前题/提交/草稿/完成/详情/报告/导出/查询/删除）。
- 服务：每个用例独立文件（`create_interview_service.go`、`submit_answer_service.go`、`list_interview_sessions_service.go` 等），便于 wire 单测替换。
- 端口（`repository/`）：
  - `InterviewSessionWriter`（写库 + 历史题 + 评估流水线状态机）
  - `InterviewSessionCache`（Redis：会话缓存 + `TryAcquireCreatingLock` SETNX）
  - `InterviewQuestionGenerator`（OpenAI / Stub 双实现）
  - `InterviewEvaluateEnqueuer`（向 `streamkey.StreamInterviewEvaluate` 入队）
  - `InterviewerRoleReader` / `ResumeTextSource`（**由 `*ResumeMapper` 同时实现**，靠 `wire.Bind` 复用）
- 状态机（`shared/interview/session_status.go`）：
  ```
  CREATED → QUESTIONS_PENDING → QUESTIONS_FAILED ┐
                              ↓                   │
                          IN_PROGRESS ────────────┤
                              ↓                   │
                           COMPLETED → EVALUATED  │
                              ↑                   │
                         (PUT 草稿不变状态) ────────┘
  evaluate_status: PENDING → PROCESSING → COMPLETED / FAILED
  ```
- 提交/草稿差异：
  - `POST /sessions/{id}/answers` → 推进游标；最后一题置 `COMPLETED` + `evaluate_status=PENDING` + 入队评估。
  - `PUT  /sessions/{id}/answers` → 仅落库当题、刷新 Redis 缓存；不改游标、不入队、允许 `answer` 空串以清空草稿。

### 6.3 知识库 / RAG

- `internal/application/knowledgebase/controller`：`POST /upload`、`GET /list`、`GET /{id}`、`GET /category/{category}`、`GET /uncategorized`、`PUT /{id}/category`、`GET /categories`、`GET /stats`、`GET /search?keyword=`、`POST /query`、`POST /query/stream`（SSE）、`DELETE /{id}`、`GET /{id}/download`（原始文件，非 JSON `Result`）、`POST /{id}/revectorize` 等已实现；上传后入队 `knowledge:vectorize:stream` 由 `StartKnowledgeVectorizeConsumer` 消费（**LLM 分片**→Embeddings→表 `knowledge_base_chunks`，维度须与显式配置的 **`KB_EMBEDDING_DIMENSIONS`** 及 `internal/db/schema/07_knowledge_base_chunks.sql` 一致）；须配置 **`KB_EMBEDDING_OPENAI_API_KEY`**、**`KB_EMBEDDING_OPENAI_BASE_URL`**、**`KB_EMBEDDING_MODEL`**、**`KB_EMBEDDING_DIMENSIONS`**（Embeddings 专用网关；无异步默认 URL）；缺失即启动 **Fatal**。分片走主 **`OPENAI_API_KEY`** + **`OPENAI_BASE_URL`** 的 Chat Completions（与简历/面试同源客户端），模型默认 **`AI_MODEL`**，可用 **`KB_CHUNK_AI_MODEL`** 等覆盖，见 `KnowledgeChunkingConfig` / `NewOpenAIKnowledgeTextChunker`。见 `EmbeddingHTTPClient`。`POST /query` / `query/stream` 走 **`KnowledgeBaseQueryService`**（`provideKnowledgeBaseEmbedder` + `KnowledgeVectorSearcher.SearchSimilarChunks` + **`KnowledgeBaseQueryChatAdapter`**）；`initializeKnowledgeBaseController(..., oa *OpenAIService)` 须注入聊天客户端。
- `internal/application/ragchat/controller` + `RagChatSessionService` + `RagChatMapper`：`/sessions` 创建/列表、`/{sessionId}` 详情、改标题/知识库/置顶/删除、`POST .../messages/stream`（SSE，`RagChatStreamService` 复用 `KnowledgeBaseQueryService.QueryStream`）已实现。表见 `internal/db/schema/06_rag_chat.sql`。
- 可观测性：`KnowledgeBaseQueryService` 在 Embedding 之后打 **`logmsg.MsgKnowledgeBaseQueryRetrieval`**（`embeddingDim`、`searchTopK`、`maxCosineDistance`、保留/丢弃计数、**`topRawHits`** 含 `chunkId` / `distance` / 正文摘要等）；**不**落完整向量。检索 SQL 在 `KnowledgeBaseMapper.SearchSimilarChunks` 中返回 `id`、`chunk_index` 以供对账。

## 7. 异步链路（Redis Streams）

### 7.1 流与组定义（`shared/streamkey/streamkey.go`）

| Stream | Group | 入队点 | 消费者 | 用途 |
|--------|-------|--------|--------|------|
| `resume:analyze:stream` | `analyze-group` | `ResumeUploadService` / `ReanalyzeResumeService` | `StartResumeAnalyzeConsumer` | 简历 LLM 分析 |
| `interview:generate-questions:stream` | `interview:generate-questions:group` | （预留）| 面试出题异步化（当前同步出题，预留接口）| 题目生成 |
| `interview:evaluate:stream` | `evaluate-group` | `SubmitAnswerService` / `CompleteSessionService` | `StartInterviewEvaluateConsumer` | 整卷评估 |
| `knowledge:vectorize:stream` | `knowledge-vectorize-group` | `UploadKnowledgeBaseService`（入队） | `StartKnowledgeVectorizeConsumer` | 知识库 **AI 分片**（`OpenAIKnowledgeTextChunker`，`OPENAI_BASE_URL` Chat JSON）、`OpenAIKnowledgeEmbedder`、写入 `knowledge_base_chunks`（pgvector）、回写 `chunk_count`+`COMPLETED`；可选 **`KB_CHUNK_AI_MODEL`** / **`KB_CHUNK_MAX_INPUT_RUNES`** / **`KB_CHUNK_MAX_COMPLETION_TOKENS`** / **`KB_CHUNK_TEMPERATURE`**（见配置校验） |

### 7.2 消费者通用模式

- 启动期 `XGROUP CREATE MKSTREAM ... 0`，遇 `BUSYGROUP` 视为已存在；运行中遇 `NOGROUP` 自动重建。
- `XReadGroup` 单批次小（`Count=4`），`Block=3s`，便于 `ctx.Done()` 在停服时快速生效。
- 处理流程：解析 → 状态机门禁（如 `evaluate_status=PENDING` 才处理）→ `TryMarkEvaluateProcessing`（CAS）→ 拉题/答案 → 调 LLM（`context.WithTimeout(15min)`）→ 持久化 → `XAck`；任一关键失败均 `MarkEvaluateFailed` 并 `XAck`（不重试，靠人工/重新入队）。
- 消费者名带 PID（`evaluate-consumer-go-<pid>`），多实例不冲突。

### 7.3 防并发（创建会话）

`CreateInterviewService` 用 `interviewSessionCache.TryAcquireCreatingLock(resumeID, 10min)` 走 Redis SETNX 互斥；与 `createInterviewWorkMax` 对齐 TTL。`workCtx = context.WithoutCancel(ctx)` 避免浏览器/反代 90 秒断连导致 LLM 与落库 `context canceled`。

## 10. 配置与环境

`internal/config/*.go` 在 `LoadEnvironmentVariables()` 一次性把所有 ENV 解析为强类型 `Config`：

| 子配置 | ENV 关键字段 |
|--------|-------------|
| `ServerConfig` | `SERVER_HOST` / `SERVER_PORT` / `SERVER_READ_TIMEOUT_SEC` |
| `DatabaseConfig` | `DATABASE_URL`（必填，校验在 `validateDatabaseConfig`） |
| `RedisConfig` | `REDIS_HOST` / `REDIS_PORT` / `REDIS_PASSWORD` / `REDIS_DB` |
| `StorageConfig` | S3/OSS endpoint/region/bucket/access keys（缺失时禁用上传依赖） |
| `OpenAIConfig` | `OPENAI_API_KEY` / `OPENAI_BASE_URL` / `AI_MODEL` / 各类 token 上限 |
| 其他 | `MAX_RESUME_UPLOAD_BYTES`（**必填**，否则 `log.Fatalf`）、`HTTP_ACCESS_LOG_SUPPRESS`（默认在代码中硬编码两条）、`LOG_LEVEL` / `DEBUG`（zap 等级） |

**禁止**在业务包内直接 `os.Getenv`；新增配置项一律走 `internal/config/*` 解析 + 落到 `Config`。

### 11.1 创建面试会话


## 12. 可观测与韧性

- 所有 INFO/ERROR 日志键走 `shared/logmsg/logmsg.go`，**不**散落字符串字面量；新加日志先在 `logmsg.go` 加常量。
- 错误文案集中在 `shared/errmsg/*.go`，前后端共用同一份提示文本；新增端点的人写错误必须先在 `errmsg/*.go` 落常量。
- HTTP 写超时 `15min`、读 header 超时 10s（`cmd/server/main.go::newHttpServer`）；导出 PDF 与同步出题路径均在 15 分钟内完成。
- 关闭：SIGINT/SIGTERM → `httpServer.Shutdown(15s)`；`StartDeps` 的 `cleanup` 逆序关闭 Redis/Postgres。
- 消费者完全独立 goroutine，停服由 `ctx.Done()` 驱动，未在 PEL 中处理的消息留在 Redis（下次启动从 `>` 继续）。

## 13. 风险与待办（来自代码现状）

1. `internal/db/` 仅为历史/空目录占位；`ai` 与 `redis/stream` 已改为依赖 `application/interview/model/results` 中的题目与报告结构，避免 `infrastructure` 反向依赖 `interfaces`。
2. `GET /api/resumes/health` 与主项目部分文档列表不一致，仓库仅提供 `GET /health`（站点级 `system` 包）。
3. 出题异步流（`StreamInterviewGenerateQuestions`）字段已在 `streamkey` 定义，但当前 `CreateInterviewService` 仍同步调 LLM；如要改为异步出题，按面试评估消费者的形状新增 `interview_questions_consumer.go` 即可。
