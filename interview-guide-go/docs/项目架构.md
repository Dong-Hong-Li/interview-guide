# 项目架构（`interview-guide-go`）

> 本文件以**当前仓库代码**为唯一事实来源（不要套主项目 Java 后端 `modules/*` 或 `router/router.go` 的目录习惯）。
> 配合阅读：[`./README.md`](./README.md)（文档索引）、`./开发规范.md`、`./项目管理.md`、`./开发进度.md`；仓库根 `README.md` 负责整个 monorepo 的总览。

---

## 1. 项目定位

- **模拟面试 + 简历分析** 的 Go 后端，与主项目 Java 后端的「面试域 + 简历域」能力对齐；**知识库**（上传 / 列表 / 分类 / 检索 / 重向量化）与 **RAG 对话**（会话 CRUD + `POST .../messages/stream` SSE）均**已实现**。
- 单进程 HTTP 服务（`cmd/server`）。**面试评估、简历 LLM 分析、知识库向量化** 走 Redis Stream 异步消费者；**面试出题** 当前仍由 `CreateInterviewService` 同步调 LLM（`streamkey.StreamInterviewGenerateQuestions` 常量已预留，未注册消费者）。**PDF 报告** 在进程内同步生成。
- 关键中间件：**PostgreSQL 16+**（GORM + `pgvector`）、**Redis 7+**（Streams + 字符串缓存 + SETNX 锁）、**对象存储**（S3 / RustFS / OSS 兼容，AWS SDK v2）、**OpenAI 兼容客户端**（`openai-go`，主对话与 Embeddings 可走两套不同网关）。

铁律性约束（在本目录全文反复出现，先在此声明）：

- **`internal/infrastructure/*` 不得依赖 `internal/interfaces/*`**（基础设施反向依赖站点壳是禁止的）。
- 与 LLM、Redis 消费者共用的题目 / 报告结构在 `internal/application/interview/model/results`（及同层 `model/`），由**应用层定义**，基础设施侧仅依赖该包或 `shared/*`。
- `internal/db/` **不是历史遗留**：包含 schema 增量 SQL、宿主机补表脚本与说明（详见本文 §4 与 [`internal/db/README.md`](../internal/db/README.md)）。

## 3. 分层与依赖方向（DDD 切片 + Hexagonal）

### 3.1 四层切片（自上而下，单向依赖）

```
interfaces (站点壳: 路由聚合、全局 mw、HTTP 绑定、SSE/PDF 二进制)
        ↓
application/<domain>/controller   ← HTTP 适配：bind / 入参校验 / 封装 Validated* / 出参 Result
        ↓                              ↑（注入 services）
application/<domain>/service       ← 用例编排（一个文件一个用例）
        ↓                              ↑（依赖 repository 端口）
application/<domain>/repository    ← 端口（interface），不依赖任何具体实现
        ↑（实现）
infrastructure/<adapter>           ← Mapper（GORM）/ Redis adapter / Storage / AI / PDF / File
```

铁律：

- **`internal/application/*` 禁止 import `internal/infrastructure/*`**（除非 controller 层组装时由 deps 显式注入）。具体实现通过 `repository.*Port`（接口）反转。
- **`internal/domain/*` 禁止 import 应用层与基础设施**：只放纯函数 / 状态常量 / 门禁，例如 `domain/interview/session_gates.go::CompleteInterviewGate`、`domain/knowledgebase/vector_status.go::IsValidVectorStatus`。
- **`shared/*` 禁止 import `internal/*`**：错误文案、日志键、Stream 键、UUID、跨域状态字面量（`shared/interview/session_status.go`）等都属此层。
- **`internal/interfaces/*` 不写业务**，只做路由聚合、全局中间件、HTTP 绑定、SSE / PDF 二进制端点的最小壳。
- **CORS**（`middleware/cors.go`，`go-chi/cors`）：必须在环境中配置 **`CORS_ALLOWED_ORIGINS`**（逗号分隔；本地前端示例 `http://localhost:5173`）；缺失则 `LoadEnvironmentVariables` 直接 **`log.Fatalf`**。也允许设为 `*`（此时不携带 Cookie）。若通过**局域网 IP** 访问 Vite（如 `http://192.168.2.1:5173`），可在 `CORS_ALLOWED_ORIGINS` 写完整源，或追加 `http://192.168.*:5173` / `http://10.*:5173`，或开 **`CORS_ALLOW_LAN_VITE_ORIGINS=1`**（由 `mergeCORSAllowedOriginsWithLanVite` 合并通配；**生产勿开**）。
- **HTTP 入参规则**（必填、Trim、MIME / 大小、multipart 正文抽取等）**只在 `application/<domain>/controller` 完成**；`service` **不校验前端原始请求体**，只接收 `model.Validated*` 或领域层已接纳的入参（详见 `./开发规范.md` §3.2、§4.3）。

## 4. 入口与依赖装配（`cmd/server`）

| 文件 | 角色 |
|------|------|
| `main.go` | `logger.NewLogger()` → `config.LoadEnvironmentVariables()` → `cfg.LogStartup(lg)` → `StartDeps(ctx, lg, cfg)` 拿到 `apiControllers + cleanupDeps` → `httpserver.RegistrationRoutes` → `newHttpServer` → `ListenAndServe`；监听 `SIGINT/SIGTERM`，`Shutdown(15s)` 后 `cleanupDeps()`。 |
| `deps.go` | **组合根**：起 `storage / postgres / redis / ai.OpenAIService` → 依次 `initialize{Resume,Interview,KnowledgeBase,RagChat}Controller`（wire 生成的 injector）→ 启动 3 个 Redis Stream 消费者 → 返回 `[]RouteRegistrar` 与逆序 `cleanup`。 |
| `wire.go` | wire 声明（`wireinject` build tag）。**4 个 `*ModuleSet`**：`resumeModuleSet` / `interviewModuleSet` / `knowledgeModuleSet` / `ragModuleSet`；接口与实现的 `wire.Bind`；`provide*` 函数处理需要从 `*Config` 抽字段或带条件 panic 的场景。 |
| `wire_gen.go` | wire 工具生成产物，**勿手改**；改完后 `wire ./cmd/server` 重新生成（`go install github.com/google/wire/cmd/wire@latest`）。 |

启动顺序与失败语义：

1. `logger.NewLogger()` 失败 → `log.Fatalf(logmsg.MsgLoggerInitFatal, ...)`；
2. `config.LoadEnvironmentVariables()` 任一子配置 `validate*` 失败 → `log.Fatalf(errmsg.LogFatalValidate*Config, ...)`（`server / database / redis / storage / openai / cors / max_resume_upload_bytes` 均在此口落地）；
3. `Storage / Postgres / Redis / OpenAI` 任一启动失败 → 仅 `lg.Error(logmsg.Msg*StartFailed, ...)` 并 `return nil, cleanup`；HTTP 仍然会启动（路由壳已挂载），但**业务控制器不会注册**，所有业务接口 404。
4. **三个 Redis Stream 消费者必须启动**：`startResumeAnalyzeConsumerIfReady` / `startInterviewEvaluateConsumerIfReady` / `startKnowledgeVectorizeConsumerIfReady` 入口都先做 `cfg / redisService / oaSvc / pg.DB` 非空校验，缺失即 **`lg.Fatal`**（禁止静默跳过，避免上传完没人消费）。
5. **面试出题**走 `provideInterviewQuestionGenerator(oa, cfg, lg)`：OpenAI 客户端不可用直接 `panic`（**无 Stub 兜底**），与历史 `stub_interview_questions.go` 仅作离线本地调试参考。
6. **Postgres DDL**：`compose.yaml` 挂载的 `docker-entrypoint-initdb.d` **仅在数据卷为空时**跑一遍。本地 `pgdata` 卷早于某次 `schema/*.sql` 入库就存在的话，须在宿主机执行 [`internal/db/apply_schema.sh`](../internal/db/apply_schema.sh)（`DATABASE_URL`）补表（最常见缺表：`knowledge_base_chunks`，详见 [`internal/db/README.md`](../internal/db/README.md)）。

`HTTP Server` 关键参数（`cmd/server/main.go::newHttpServer`）：

| 参数 | 值 | 含义 |
|------|----|------|
| `ReadHeaderTimeout` | 10s | 慢 header 防御 |
| `ReadTimeout` | `cfg.Server.ServerReadTimeoutSec` | 含 body 的总读超时 |
| `WriteTimeout` | **15min** | 同步出题 / PDF 导出 / RAG SSE 都必须在 15 分钟内写完 |
| `IdleTimeout` | 120s | keep-alive |

## 5. HTTP 路由与请求生命周期

### 5.1 全局壳 `internal/interfaces/controller/router.go`

```text
chi.NewRouter()
  ├─ Use(middleware.CORS(cfg))    # 必须在最外层，否则 OPTIONS 预检会被后续中间件阻断
  ├─ Use(chimw.RequestID)
  ├─ Use(chimw.RealIP)
  ├─ Use(chimw.Recoverer)
  ├─ Use(middleware.RequestLogger(lg, cfg.HTTPAccessLogSuppress))
  │
  ├─ (&rootController{}).Register(r)
  │     ├─ GET /         → text/plain "interview-guide-go\n"
  │     └─ GET /health   → JSON {"status":"ok"}
  │
  └─ Route("/api"):
        ├─ (&apiMetaController{}).Register(api)   # GET /api/meta → 空 {}
        └─ for reg := range registrars: reg.Register(api)
              # ResumeController / InterviewController /
              # KnowledgeBaseController / RagChatController
```

注意：当前**没有**显式的 `404 / 405` JSON 钩子，全部走 chi 默认响应。如需统一返回 `Result` 结构，须在本文件后续 PR 中重新挂 `r.NotFound` / `r.MethodNotAllowed`。

`RouteRegistrar` 接口：

```go
type RouteRegistrar interface { Register(r chi.Router) }
```

各域控制器在自己的 `Register(api)` 内部 `api.Route(APIMountPath, ...)` 嵌套子路由（`APIMountPath` 集中在 `application/<domain>/controller/api.go` 的常量里，例如 `/resumes`、`/interview`、`/knowledgebase`、`/rag-chat`）。**业务接口统一挂在 `/api` 下**，不使用 `/api/v1` 前缀。

### 5.2 单请求处理链路（`internal/interfaces/http/binding`）

`binding` 包提供**两个泛型入口**：

| 入口 | 适用场景 | 行为 |
|------|----------|------|
| `Handle[Req, Resp any](fn func(ctx, Req) (Resp, error), opts ...Option)` | 绝大多数端点 | `bindRequest` → `Validate` → `fn` → `WriteJSON Result.Success(resp)` |
| `Exec[Resp any](fn func(ctx) (Resp, error))` | 无入参 GET（健康检查、静态枚举） | 直接 `fn` → `WriteJSON Result.Success(resp)` |

`bindRequest` 行为：

- `POST/PUT/PATCH` 时按 `Content-Type` 选 JSON 或 multipart：
  - `application/json`：`http.MaxBytesReader` 限到 `MaxBody`（默认 **4 MiB**），`json.Decoder.Decode`；空体 → 400；超大 → 413；解析失败 → 400。
  - `multipart/form-data`：`ParseMultipartForm(MultipartMaxMemory)`（默认 **32 MiB**）→ `bindMultipart`：按 `form` 标签把 `[]byte` 视为文件体，`string` 字段走 `FormValue + TrimSpace`；同时在结构体中有 `Filename` / `ContentType` 字段时从 `FileHeader` 回填。
- 之后调 `bindParams`：按 `path:"name"` 标签从 `chi.RouteContext` 取，按 `query:"name"` 标签从 `URL.Query()` 取，并自动 `setField` 到 `string / int* / uint* / bool / float* / *T`。
- `MaxBody(n)` / `MultipartMaxMemory(n)` 是 `Handle` 的可选 `Option`，按域覆盖默认值。

`Validate` 行为：

- 反射读 `validate:"required"`（含子串 `required` 即可）。零值即 400；`[]byte` 视为文件体时空切片视为缺失。
- 错误消息优先取 `json` / `path` / `query` / `form` 标签名，否则用 `field.Name`。
- **`int = 0` 也视为零值**，需要严格区分「未传 / 0」时改用 `*int`。

成功 / 失败统一出口（`shared/response`）：

- 成功：`WriteJSON 200 + Result{ code:200, message:"success", data:resp }`。
- `*response.Error`（来自 `response.Err(code, msg)`）：HTTP 状态码与 `Result.code` 一致（`writeAsHTTPResult` 走 `ErrJSON`）。
- `*response.BizError`（来自 `response.BizErr(code, msg)`）：HTTP=200，但 `Result.code` 是业务码（用于客户端弱通道展示）。
- 其他 `error` → `response.WriteErr` → 500 + `errmsg.InternalServerError`。

### 5.3 PDF / 二进制端点

不走 `binding`，直接用 `chi.URLParam` + `response.ErrJSON / WriteErr`，写头：

| 端点 | 头 | 文件名兼容 |
|------|----|-----------|
| `GET /api/resumes/{id}/export` | `Content-Type: application/pdf` | `pdfexport.ContentDispositionRFC5987(filename)` |
| `GET /api/interview/sessions/{sessionId}/export` | 同上 | `ReportService.ExportInterviewPDF` 直接返回 `disposition` 字符串 |
| `GET /api/knowledgebase/{id}/download` | `Content-Type: <存原 content_type>` | `pdfexport.ContentDispositionRFC5987(filename)` |

### 5.4 SSE 流式端点

| 端点 | 实现入口 | 备注 |
|------|----------|------|
| `POST /api/knowledgebase/query/stream` | `KnowledgeBaseController.handleQueryKnowledgeBaseStream` → `KnowledgeBaseQueryService.QueryStream` | 自己 `Decode` + `Validate`，直接写 `text/event-stream`，内部按需 `flush()` |
| `POST /api/rag-chat/sessions/{sessionId}/messages/stream` | `RagChatController.handleSendMessageStream` → `RagChatStreamService.StreamSessionMessage`（复用 `KnowledgeBaseQueryService.QueryStream`） | 同上；写超时与 `WriteTimeout=15min` 匹配 |

**SSE 响应头（统一）**：`Content-Type: text/event-stream; charset=utf-8`、`Cache-Control: no-cache`、`Connection: keep-alive`。`Flusher` 不可用时退化为缓冲写。

### 5.5 访问日志抑制（`config.parseHTTPAccessLogSuppress`）

代码内**硬编码**抑制规则（`middleware.ShouldSuppressAccessLog` 用 `path.Match` 评估）：

- `GET /api/interview/sessions/*`（前端答题页轮询单会话）
- `GET /api/resumes/*/detail`
- `GET /api/knowledgebase/list`
- `GET /api/knowledgebase/stats`
- `GET /api/knowledgebase/categories`

**新增高频轮询接口须同步加规则**，避免日志噪声；规则当前不走 ENV，要在 `config.go::parseHTTPAccessLogSuppress` 里改。

## 7. 异步链路（Redis Streams）

### 7.1 流与组定义（`shared/streamkey/streamkey.go`）

| Stream | Group | 入队点 | 消费者 | 用途 |
|--------|-------|--------|--------|------|
| `resume:analyze:stream` | `analyze-group` | `ResumeUploadService` / `ReanalyzeResumeService` | `StartResumeAnalyzeConsumer` | 简历 LLM 分析 |
| `interview:generate-questions:stream` | `interview:generate-questions:group` | **暂未入队** | **暂未启动** | 预留：面试出题异步化（当前 `CreateInterview` 同步出题；如要异步化按面试评估消费者形状新增 `interview_questions_consumer.go`） |
| `interview:evaluate:stream` | `evaluate-group` | `SubmitAnswerService` / `CompleteSessionService` | `StartInterviewEvaluateConsumer` | 整卷评估 |
| `knowledge:vectorize:stream` | `knowledge-vectorize-group` | `UploadKnowledgeBaseService` / `RevectorizeKnowledgeBaseService` | `StartKnowledgeVectorizeConsumer` | 知识库 **AI 分片**（`OpenAIKnowledgeTextChunker`，主对话 Chat JSON）→ `OpenAIKnowledgeEmbedder` Embedding → 写入 `knowledge_base_chunks`（pgvector）→ 回写 `chunk_count` + `vector_status=COMPLETED`。可选：**`KB_CHUNK_AI_MODEL`** / **`KB_CHUNK_MAX_INPUT_RUNES`** / **`KB_CHUNK_MAX_COMPLETION_TOKENS`** / **`KB_CHUNK_TEMPERATURE`** |

知识库向量化的 Embeddings **走独立网关**：必须配置 **`KB_EMBEDDING_OPENAI_API_KEY`**、**`KB_EMBEDDING_OPENAI_BASE_URL`**、**`KB_EMBEDDING_MODEL`**、**`KB_EMBEDDING_DIMENSIONS`**；缺失 → `lg.Fatal(logmsg.MsgKnowledgeEmbeddingClientFatal, ...)`。**`KB_EMBEDDING_DIMENSIONS`** 与 `internal/db/schema/07_knowledge_base_chunks.sql` 中 `vector(N)` 列的 `N` **必须严格一致**；调整维度需走 `migrate_kb_chunks_embedding_to_1024.sql` 这类迁移脚本。分片 / 主对话 / 简历 / 面试共享主网关 **`OPENAI_API_KEY`** + **`OPENAI_BASE_URL`** + **`AI_MODEL`**。

### 7.2 消费者通用模式（`internal/infrastructure/redis/stream/*.go`）

- 启动期 `XGROUP CREATE MKSTREAM ... 0`，遇 `BUSYGROUP` 视为已存在；运行中遇 `NOGROUP` 自动重建。
- `XReadGroup` 单批次小（`Count=4`），`Block=3s`，便于 `ctx.Done()` 在停服时快速生效。
- 处理流程：解析 → 状态机门禁（如 `evaluate_status=PENDING` 才处理）→ `TryMarkEvaluateProcessing`（CAS）→ 拉题 / 答案 / 简历正文 → 调 LLM（`context.WithTimeout(15min)`）→ 持久化 → `XAck`；任一关键失败均 `MarkEvaluateFailed` 并 `XAck`（**不重试**，靠人工 / 重新入队）。
- 消费者名带 PID（如 `evaluate-consumer-go-<pid>`），多实例不冲突。

### 7.3 防并发（创建会话）

`CreateInterviewService` 用 `interviewSessionCache.TryAcquireCreatingLock(resumeID, 10min)` 走 Redis SETNX 互斥；与 `createInterviewWorkMax`（同步出题最大耗时）对齐 TTL。`workCtx = context.WithoutCancel(ctx)` 隔离请求 ctx，避免浏览器 / 反代 90 秒断连导致 LLM 与落库 `context canceled`。

## 12. 可观测与韧性

- **日志键统一**：所有 INFO/ERROR 日志键走 `shared/logmsg/logmsg.go`，**不**散落字符串字面量；新加日志先在 `logmsg.go` 加常量。
- **错误文案统一**：`shared/errmsg/*.go`（按域拆 `common / config / interview / knowledgebase / ragchat / resume`），前后端共用同一份提示文本；新增端点的人写错误必须先在 `errmsg/*.go` 落常量。
- **超时**：HTTP `WriteTimeout=15min`、`ReadHeaderTimeout=10s`；同步出题 / 整卷评估 / KB 分片 / RAG SSE 内部 LLM 调用都用 `context.WithTimeout(15min)`。
- **关停**：`SIGINT/SIGTERM` → `httpServer.Shutdown(15s)`；`StartDeps` 的 `cleanup` **逆序**关闭 Redis / Postgres。消费者完全独立 goroutine，停服由 `ctx.Done()` 驱动；未在 PEL 中处理的消息留在 Redis（下次启动从 `>` 继续）。
- **CORS**：`CORS_ALLOWED_ORIGINS` 必填；缺失即启动 Fatal。生产环境**勿开** `CORS_ALLOW_LAN_VITE_ORIGINS`。
