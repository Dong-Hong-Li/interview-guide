# 开发规范（`interview-guide-go`）

> 本规范以**当前仓库**已落地的写法为唯一参照（见 `internal/...` 实际代码与 `docs/项目架构.md`）。  
> 新增代码不得偏离本规范；若必须偏离，先改本规范再改代码。

---

## 1. 通则

- **语言/版本**：Go `1.24+`（见 `go.mod`），不依赖 cgo（`Dockerfile` 中 `CGO_ENABLED=0`）。
- **风格**：先 `gofmt`/`gofumpt`，再 `goimports`；提交前 `go vet ./...` 与 `go build ./...` 通过。
- **PR 大小**：≤ 400 行变更；≥ 800 行须拆分。
- **可观测性**：能落 INFO 的就落 INFO，关键失败必须落 ERROR；所有日志走 zap + `shared/logmsg` 常量键。
- **零环境字面量**：业务包内**禁止**出现 `os.Getenv`，全部在 `internal/config/*` 解析。
- **零中文字面量散落**：错误文案 / 日志 message 必须放 `shared/errmsg` 或 `shared/logmsg`，业务代码只引用常量。

## 2. 包与目录

### 2.1 命名

| 层级 | 包名 | 目录 |
|------|------|------|
| 控制器 | `controller`（小写单数） | `internal/application/<domain>/controller` |
| 应用服务 | `service` | `internal/application/<domain>/service` |
| 仓储端口 | `repository` | `internal/application/<domain>/repository` |
| 入参/出参 | `model`（含 `model/results` 子包） | `internal/application/<domain>/model` |
| 领域 | `<domain>`（如 `interview`） | `internal/domain/<domain>` |
| 适配器 | 平实命名（`mapper`/`stream`/`adapter`/`promptprofile`/`pdf`/...） | `internal/infrastructure/<group>/<adapter>` |
| 站点壳 | `httpserver`（`controller/router.go`）/ `binding` / `middleware` | `internal/interfaces/...` |

> 跨域共享放 `shared/<topic>`（如 `shared/interview/session_status.go`、`shared/streamkey/streamkey.go`）。**禁止**反向 import：`shared` 不依赖 `internal`。

### 2.2 文件粒度

- 应用服务：**一个用例一文件**（参考 `service/create_interview_service.go`、`submit_answer_service.go`、`list_interview_sessions_service.go`）。
- 控制器：**一个域一文件**，`controller.go` 列全部 handler，`api.go` 集中放路径常量。
- Mapper：**一个聚合根一文件**（`interview_mapper.go`、`resume_mapper.go`）；新方法贴在同文件末尾，并在文件头 `var _ repository.X = (*XMapper)(nil)` 自检。
- 单元测试：与被测文件同目录 `xxx_test.go`，包名 `<pkg>_test`（黑盒）或 `<pkg>`（白盒，仅在确实需要内部访问时）。

### 2.3 import 分组（用 goimports 强制）

```go
import (
    // 1. 标准库
    "context"
    "net/http"

    // 2. 第三方
    "github.com/go-chi/chi/v5"
    "go.uber.org/zap"

    // 3. 本仓库
    "interview-guide-go/internal/application/interview/model"
    "interview-guide-go/shared/response"
)
```

**禁止** `dot import` 与 `_` 匿名 import（除非 driver 注入，目前没有此用法）。

## 3. HTTP 接口规范

### 3.1 路由注册

```go
// internal/application/<domain>/controller/api.go
const (
    APIMountPath          = "/interview"
    PathSessionAnswers    = "/sessions/{sessionId}/answers"
    // ... 路径片段全部以 const 形式集中
)

// controller.go
func (c *XController) Register(r chi.Router) {
    r.Route(APIMountPath, func(sr chi.Router) {
        sr.Post(PathSessionAnswers, binding.Handle(c.submitAnswer))
        sr.Put(PathSessionAnswers, binding.Handle(c.saveAnswer))
        // ...
    })
}
```

- 业务接口**统一挂在 `/api`** 之下，由 `internal/interfaces/controller/router.go` 在 `r.Route("/api", ...)` 内逐个调 `m.Register(r)`。
- **不**自定义 `/api/v1` 等版本前缀；版本演进通过资源命名或字段扩展处理。
- 路径风格 `/<resource>/{id}/<sub-resource>`（小写连字符）；动词写在 HTTP method，不写在路径中（`/.../complete` 这类「显式动作」是例外，需在 `api.go` 注释说明）。

### 3.2 入参与绑定

- 一律使用泛型包装 `binding.Handle[Req, Resp](fn)` 或 `binding.Exec[Resp](fn)`（无入参时）。**不要**裸写 `func(w http.ResponseWriter, r *http.Request)`，唯一例外是 PDF 二进制端点（必须自管 `Content-Type` / `Content-Disposition`）。
- 入参结构体放 `application/<domain>/model/request_param.go`，字段标签按用途选用：
  - `json:"xxx"` —— application/json body
  - `path:"xxx"` —— chi path param（与 `/{xxx}` 同名）
  - `query:"xxx"` —— URL query
  - `form:"xxx"` —— multipart/form-data；`[]byte` 字段视为文件，结构体内同名 `Filename`/`ContentType` 会被自动填充
  - `validate:"required"` —— 反射校验：零值即 400
- **整型 0 与未传无法区分**：必填 0 字段须改成 `*int` / `*int64`（参考 `CreateInterviewSessionReq.ResumeID *int64`）。
- **application/service 层不对前端/HTTP 传入的原始请求做校验**（如 `Trim`、非空、 MIME、文件大小、正文是否可解析等）；这些**全部在 controller 层**完成。通过后封装为 `model.Validated*`（参考 `ValidatedSubmitAnswer`、`ValidatedKnowledgeBaseUpload`）再传入 service。service 只做**领域规则**与编排（见 §4.3），不重复 HTTP 维度的校验。
- 路径参数为字符串 ID 时仍用 `path` 标签 + 控制器 `strings.TrimSpace` 双检（参考 `submitAnswer`）。

#### 3.2.1 知识库域（`application/knowledgebase`）示例：上传

- **Service**（`UploadKnowledgeBaseService`）**只接收** `model.ValidatedKnowledgeBaseUpload`，**不**接触 `KBPostUploadRequest`，**不**做入参校验与正文抽取。
- **Controller**：对 `KBPostUploadRequest` 调 `binding.Validate`（`validate` 标签）、补充 `domain/knowledgebase.ValidateFile` / `ValidateContentType`、用注入的 `KnowledgeTextExtractor` **抽取正文**、空名时用 `domain/knowledgebase.DisplayNameFromFilename`，再组装 `ValidatedKnowledgeBaseUpload` 调 service。

### 3.3 出参与错误

- 成功统一 `response.Result{Code:200, Message:"success", Data:resp}`，由 `binding.Handle` 自动包装。
- 错误**只能**返回三类：
  1. `response.Err(http.StatusXxx, msg)` —— HTTP 错误（HTTP 状态码 == `Result.Code`）；
  2. `response.BizErr(code, msg)` —— 业务错（HTTP 200 + 业务码，前端继续渲染流程时使用）；
  3. 其他 `error` —— 兜底 500 + `errmsg.InternalServerError`。
- **禁止**裸 `panic`、`os.Exit`；启动期的致命问题用 `log.Fatalf`（仅 `cmd/server`、`internal/config`）。
- 错误**消息文案**全部走 `shared/errmsg/*.go` 常量；不要在 service 里写中文字面量。
- 状态码选择参考主项目（与 Java 端对齐）：
  - 入参错 → 400
  - 未鉴权 / 鉴权失败 → 401 / 403
  - 资源不存在 → 404 / 410
  - 业务冲突（如锁） → 409
  - 服务未就绪（注入项为 nil） → 503
  - 上游 LLM 失败 → 502
  - 内部异常 → 500

### 3.4 PDF / 二进制响应

- 直接 `http.Handler` 注册（参考 `handleExportInterviewPDF`）；
- `Content-Type: application/pdf`，`Content-Disposition` 用 `pdfexport.ContentDispositionRFC5987(name)`；
- 错误仍用 `response.WriteErr(w, err)`，保持错误体一致（JSON Result）。

## 4. 应用服务

### 4.1 命名

`<动作><聚合>Service`（`CreateInterviewService` / `SubmitAnswerService` / `ListInterviewSessionsService`）。一个 service 对应一个用例；超过 200 行或同时承担两个用例必须拆。

### 4.2 构造函数

- 必须叫 `New<X>Service(...) *XService`，参数顺序与 `wire.NewSet` 中 provider 顺序一致。
- 依赖**只通过端口接口**（`repository.*Port`），**不直接** import `infrastructure/*`。
- 单元测试通过 fake 实现端口，不需要起 Redis/Postgres。

### 4.3 入参契约

- 公开方法**只接收** `model.Validated*` 或基础类型（如 `(ctx, page, size int)`）。
- 不在 service 内做 HTTP 状态码以外的「重复格式校验」（`controller` 已校验 `required`/`Trim`/`下界`）；service 只做**领域规则**校验（如题号上界依赖题库长度，必须在 service 内做）。

### 4.4 上下文

- 默认沿用 `r.Context()`；
- 但**长跑路径**（出题、评估、PDF）用 `context.WithoutCancel(ctx)` + `context.WithTimeout(...)` 解绑客户端取消，避免反代 ~90s 断连导致 LLM/落库 `context canceled`（参考 `CreateInterviewService.CreateInterview` 与消费者）。
- 释放锁/收尾的 cleanup 子 ctx **必须** `WithoutCancel(ctx)` 再 `WithTimeout(5s)`。

### 4.5 错误传播

- service 直接 `return response.Err(...)` 让 controller 透传；
- 仅在边界（gorm 错误 → 业务错误）做翻译，且 `errors.Is(err, gorm.ErrRecordNotFound)` 必须显式分支。

## 5. 仓储端口

### 5.1 端口位置

`internal/application/<domain>/repository/*.go`，**接口**即定义即文档。每文件单一职责（写库 / 缓存 / 入队 / 只读 source）。

### 5.2 命名

- 写库：`<聚合>Writer`（`InterviewSessionWriter`）。
- 只读：`<能力>Reader`（`InterviewerRoleReader`）或 `<X>Source`（`ResumeTextSource`）。
- 缓存：`<聚合>Cache`（`InterviewSessionCache`）。
- 入队：`<动作>Enqueuer`（`InterviewEvaluateEnqueuer`）。
- 题库生成器：`<能力>Generator`（`InterviewQuestionGenerator`）。

### 5.3 方法签名

- 第一参数恒为 `ctx context.Context`；
- 返回值最后一项恒为 `error`；
- 入参/出参类型恒为 `application` 层模型（`model.*` / `model/results.*`），**禁止**出现 `gorm/grom` 模型或 `redis.Cmd` 等基础设施类型；
- 多返回值用具名结构体或元组（如 `([]model.Item, int64, error)` 列表 + total）。

### 5.4 实现绑定

- Mapper / Adapter 在文件末尾加编译期断言：
  ```go
  var _ repository.InterviewSessionWriter = (*InterviewMapper)(nil)
  ```
- 同一适配器满足多端口时（如 `*ResumeMapper` 同时实现 `InterviewerRoleReader` 与 `ResumeTextSource`），用 `wire.Bind` 绑定即可，**不要**新建包装类。

## 6. 基础设施适配器

### 6.1 GORM 模型

- 放 `internal/infrastructure/postgres/grom/`（**不要**改成 `gorm`，沿用现有目录名）；
- 一表一文件；字段注释中文，列名走 `column:xxx`；
- `TableName()` 显式指定；时间字段在 `BeforeCreate` 内补默认值；
- 复杂枚举（status、evaluate_status）用 `shared/interview/session_status.go` 中的常量比较，**不**写魔法字符串。

### 6.2 Mapper

- 包名 `mapper`；类型 `XMapper struct{ gdb *gorm.DB }`；
- 任何方法第一行 `if m.gdb == nil { return errors.New("xxx: nil db") }`（防注入失败时 NPE）；
- 复杂查询用 `Raw(...).Scan(...)`，列名硬编码或用 `gorm:"column:..."` 反射，避免 GORM 自动改名歧义；
- 事务用 `m.gdb.WithContext(ctx).Transaction(func(tx *gorm.DB) error { ... })`，**不**使用全局 db 句柄。

### 6.3 Redis 适配器

- 缓存 key 用前缀常量（在适配器自身包内，`session:<id>`、`interview:generate:lock:<resumeId>`）；
- Stream key / group / 字段名**必须**走 `shared/streamkey/streamkey.go`；
- SETNX 锁：`SetNX(ctx, key, "1", ttl)`；释放用 `Del`，**不**用 `EXPIRE` 续命，TTL 与业务总超时对齐。

### 6.4 消费者

- 文件 `<scenario>_consumer.go`；启动函数 `Start<Scenario>Consumer(ctx, rdb, ...deps)`；
- 必有 `ensure<X>Group` 处理 `BUSYGROUP` / `NOGROUP`；
- 单条消息处理函数命名 `process<Scenario>Message`，内部局部 `ack := func() { _ = rdb.XAck(...).Err() }`；
- 处理失败必须 `MarkXxxFailed(...)` 后再 `XAck`，**不**让消息无限循环。

### 6.5 AI 客户端

- 包 `internal/infrastructure/ai`；OpenAI 客户端工厂 `openai_service.go::NewOpenAIService`；启动失败返回 nil + log，**不**panic；
- 业务侧通过端口（`ai/adapter`）注入；prompt 模板放 `prompts/`，新角色加 `promptprofile`。

## 7. 配置与启动

- 新增配置项流程：
  1. 在 `internal/config/<topic>_config.go` 加结构体字段 + `validate<X>Config()`；
  2. `LoadEnvironmentVariables()` 内调用 + 写入 `Config`；
  3. `LogStartup` 中追加脱敏日志（`shared/logmsg.FieldXxx`）；
  4. 在本规范的「配置」一节追加 ENV 列表说明（同步 `docs/项目架构.md` §10）。
- **必填**配置缺失时直接 `log.Fatalf`（参考 `parseMaxResumeUploadBytes`），不要 fall back 到默认值掩盖问题。
- `StartDeps` 内启动顺序：Storage → Postgres → Redis → OpenAI → wire 控制器 → 异步消费者；任意基础设施失败均返回 `nil, cleanup` 让 main 提前退出。

## 8. 依赖注入（Wire）

- 所有控制器与服务**必须**通过 `cmd/server/wire.go` 中的 ProviderSet 装配；不要手写 `New*Service(New*Mapper(db), ...)` 链。
- 增加 service / repository 的步骤：
  1. 在 `application/<domain>/repository/*.go` 写端口；
  2. 在 `application/<domain>/service/*.go` 写实现；
  3. 在 `cmd/server/wire.go` 对应 `<domain>ModuleSet` 内追加 `service.New<X>Service` 与必要的 `wire.Bind`；
  4. 在 controller 结构体里加字段；
  5. 终端执行 `wire ./cmd/server` 重新生成 `wire_gen.go`，并 `go build ./cmd/server` 验证。
- `wire.Struct(new(controller.X), "*")` 用于控制器；其内部字段必须**全部**为已 provided 的接口/类型。
- **接口同名实现冲突**走 `wire.Bind(new(IFace), new(*Impl))`；不要在 set 内重复 `provideX` 函数。
- 对 nil 兜底场景（如 OpenAI 客户端不可用退回 Stub）写专门的 `provideXxx` 函数（参考 `provideInterviewQuestionGenerator`）。

## 9. 错误与日志

### 9.1 错误文案

```go
// shared/errmsg/interview.go
const SubmitAnswerSessionNotFound = "面试会话不存在"

// service
return nil, response.Err(http.StatusNotFound, errmsg.SubmitAnswerSessionNotFound)
```

- 中文为前后端共用文案；纯内部日志的英文短语可在 `shared/logmsg` 内追加为 `Msg*` 常量。
- 错误**不带敏感信息**（不拼 SQL、不拼请求体）。

### 9.2 日志

```go
lg.Info(logmsg.MsgInterviewEvaluateAIBegin,
    zap.String(logmsg.FieldSessionID, sid),
    zap.Int("questionCount", len(qs)),
)
```

- **message** 走 `logmsg.Msg*` 常量；
- **zap.Field key** 走 `logmsg.Field*` 常量（`FieldSessionID` / `FieldAIDuration` 等）；新加键必须先在 `logmsg.go` 落常量；
- 不输出 PII / 简历正文；调用模型时只记录 `model`、耗时、token 数。
- 关闭/启动事件打 INFO；预期失败打 WARN；不可恢复的内部错误打 ERROR；ERROR 自动带堆栈（`zap.AddStacktrace(zapcore.ErrorLevel)`）。

### 9.3 访问日志抑制

- 频繁轮询 / 无业务价值的 GET 端点，加规则到 `parseHTTPAccessLogSuppress`，模式语法见 `path.Match` 与 `ParseAccessLogSuppressSpec` 注释。

## 10. 数据校验与边界

- **所有外部输入** Trim 后再校验（`strings.TrimSpace`）；
- 题号、分页等数值字段：controller 检查下界（≥0/≥1），service 检查上界（依赖业务上下文，如题库长度）；
- 分页用 `internal/domain.NormalizeListPaging(page, size)`：默认 size=20，max=100；新写分页 service 必须调用一次。
- UUID 用 `shared/uuid.NewUUID16()`（16 字符短 ID）；不要直接 `uuid.New().String()`。

---

## 11. 状态机与门禁

- **唯一**真理：`shared/interview/session_status.go` 与 `internal/domain/interview/session_status.go`；
- 域内状态判断**必须**通过门禁函数（`CompleteInterviewGate`、`Status.IsCompletedOrEvaluated()` 等），不要在 service / Mapper 散落 `if status == "COMPLETED"` 字符串比较；
- 新增一个状态：先改 `shared/interview/session_status.go` 常量 → 再改 `internal/domain/interview/session_status.go` 方法 → 最后改 service / Mapper。

## 13. SQL 与 Schema

- 表/列命名 snake_case；时间字段 `created_at` / `updated_at` / `completed_at` 一律 `timestamptz`；
- 大文本字段（`questions_json`、`overall_feedback`、`*_json` 评估快照）使用 `text`；
- 外键命名 `<对端单数>_id`；本仓库现有特例：`interview_answers.session_id` 指 `interview_sessions.id`（**主键**）而非对外 `session_id` 字符串，必须在新加查询时延续此约定，并在 Mapper 注释中重申；
- 索引：高频读字段（`session_id`、`status`、`resume_id`、`created_at`）建 BTree；不依赖部分索引/表达式索引；
- 迁移：放在仓库 `schema/` 或主项目对应迁移目录（与主项目对齐），新表在 `grom/` 加模型 + Mapper 同步。

## 14. 安全

- 所有 multipart 上传走 `binding.Handle` 默认 32MB 上限，文件大小校验在 service 内（`MAX_RESUME_UPLOAD_BYTES`）；
- 不在日志/错误体中回显完整文件名、外发 URL；
- 对象存储 key 使用受控前缀 `<domain>/<id>/<uuid>.<ext>`，不允许用户自选；
- LLM 输出**不直接返回前端**，必须落库后再透出（避免 SSRF / prompt 注入回流到客户端）；
- 错误消息不暴露内部实现（栈、SQL）。
