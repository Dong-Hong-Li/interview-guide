### 4.1 知识库 + RAG 业务实现顺序（建议）

**原则**：

1. 先**可管理的数据面**（元数据、文件、分类、删除），再**检索与对话**；先有「存什么、属谁、怎么删改」，再谈 embedding 与流式回答。
2. 先**同步类接口**（`list` / `detail` / `download` / `delete` 等），后**流式**（`query/stream`、RAG 消息流），便于联调与验收。
3. **RAG 对话**依赖「可选中的知识库」与「可检索的向量」：在知识库**核心可检索**具备后再全量做 `ragchat`。

**阶段 0：设计与基础设施**（可与 D1 并行）

| 项 | 说明 |
|----|------|
| 领域模型 | 知识库文档元数据：标题、类型、大小、用户/租户、分类、`status`（上传中/可检索/失败）等。 |
| 存储 | 对象存储放原始文件（可复用简历上传相关能力）、Postgres 存元数据与分类。 |
| 向量 | 选型（如 pgvector / 外置库）、与现有 OpenAI/兼容 embedding 的维度与索引策略一致。 |
| 切分 | 纯文本 / PDF 的 chunk 规则与 `POST .../revectorize` 行为一致。 |

**验收**：能描述「从上传到可检索」的状态机，以及失败与重试策略。
